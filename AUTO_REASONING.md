# Auto 推理模式

## 概述

Auto 推理模式是一个智能系统，使用 **LLM 自分析** 来判断任务难度和类别，并自动调整推理参数以获得最佳效果。

## 工作原理

### 1. LLM 自分析

当你发送消息时，系统会：

1. **提取你的输入**
2. **调用 LLM 分析任务**（使用轻量级请求，仅 200 tokens）
3. **解析分析结果**（难度、类别、原因）
4. **自动调整推理参数**
5. **使用调整后的参数发送实际请求**

```
用户输入 → LLM 分析 → 参数调整 → 实际请求
     ↓
  "请帮我实现一个 React 组件"
     ↓
  LLM 分析: 中等难度、代码类别
     ↓
  调整参数: Temp=0.5, TopP=0.6, 推理=平衡
     ↓
  使用调整后的参数发送请求
```

### 2. 分析过程

LLM 会分析你的输入，返回以下信息：

```json
{
  "difficulty": "medium",
  "category": "code",
  "reasoning": "用户请求实现 React 组件，涉及状态管理和事件处理，属于中等复杂度的编程任务"
}
```

### 3. 参数映射

根据分析结果，系统会自动调整以下参数：

| 难度 | 类别 | Temperature | Top P | 推理强度 | Max Tokens |
|------|------|-------------|-------|----------|------------|
| 简单 | 对话 | 0.7 | 0.9 | 快速 | 2048 |
| 中等 | 代码 | 0.5 | 0.6 | 平衡 | 4096 |
| 复杂 | 分析 | 0.3 | 0.5 | 深度 | 8192 |
| 复杂 | 数学 | 0.0 | 0.1 | 深度 | 8192 |
| 中等 | 创意 | 1.0 | 1.0 | 平衡 | 4096 |
| 中等 | 写作 | 0.6 | 0.8 | 平衡 | 8192 |

## 使用方法

### 1. 启用 Auto 模式

1. 打开设置 (⚙️)
2. 选择"模型与工具配置"
3. 在"推理强度"部分选择"🤖 Auto"

### 2. 自动工作

当你输入消息时：

1. **状态栏显示分析中**
   - "🤖 正在分析任务..."

2. **控制台输出分析结果**
   ```
   [AutoReasoning] Task analysis: {
     difficulty: 'medium',
     category: 'code',
     confidence: 0.85,
     reasoning: '用户请求实现 React 组件，属于中等复杂度的编程任务'
   }
   ```

3. **自动应用调整后的参数**

## 优势

### 对比传统关键词匹配

| 特性 | 关键词匹配 | LLM 自分析 |
|------|-----------|-----------|
| 准确性 | 中等 | 高 |
| 上下文理解 | ❌ | ✅ |
| 语义理解 | ❌ | ✅ |
| 响应速度 | 快 | 稍慢 |
| Token 消耗 | 无 | 少量 |
| 维护成本 | 高 | 低 |

### 示例对比

**输入**: "我想写一个函数，它应该能够处理各种边界情况，并且要有良好的错误处理"

**关键词匹配**:
- 检测到: "函数" → 中等难度、代码类别
- 未检测到: 复杂度指示

**LLM 分析**:
- 分析: 用户强调"边界情况"和"错误处理"，表明对代码质量有高要求
- 判断: 中等偏难、代码类别
- 原因: 需要考虑多种边界情况，编写健壮的代码

## 技术实现

### 核心函数

```typescript
// 使用 LLM 分析任务
async function analyzeTaskWithLLM(
  input: string,
  callLLM: (prompt: string) => Promise<string>
): Promise<TaskAnalysis>

// 快速分析（fallback）
function analyzeTaskQuick(input: string): TaskAnalysis

// 应用自动推理
async function applyAutoReasoning(
  input: string,
  currentParams: Partial<ReasoningParams>,
  callLLM?: (prompt: string) => Promise<string>
): Promise<{ params: ReasoningParams; analysis: TaskAnalysis }>
```

### API 调用优化

为了减少延迟，LLM 分析使用：
- **低温度**: 0.1（确保稳定输出）
- **小 token 限制**: 200 tokens（减少消耗）
- **轻量级 prompt**: 专门设计的分析 prompt

### Fallback 机制

如果 LLM 分析失败，系统会：
1. 使用快速规则分析
2. 使用默认参数
3. 记录错误日志

## 性能考虑

### 延迟分析

- LLM 分析请求: ~200-500ms
- 参数调整: <1ms
- 总延迟: ~200-500ms

### Token 消耗

- 分析请求: ~100 tokens 输入 + ~50 tokens 输出
- 总消耗: ~150 tokens/次

### 优化建议

1. **缓存分析结果**
   - 相似输入可以复用分析结果
   - 减少重复的 LLM 调用

2. **异步分析**
   - 在等待分析时预加载其他资源
   - 减少用户感知的延迟

3. **批量分析**
   - 对于多条消息，可以批量分析
   - 减少 API 调用次数

## 故障排除

### LLM 分析失败

**症状**: 使用默认参数，控制台显示错误

**解决方案**:
1. 检查 API 配置是否正确
2. 检查网络连接
3. 查看错误日志

### 分析结果不准确

**症状**: 调整后的参数不适合当前任务

**解决方案**:
1. 手动覆盖参数
2. 调整分析 prompt
3. 使用更强大的模型

### 延迟过高

**症状**: 发送消息后等待时间长

**解决方案**:
1. 使用更快的模型进行分析
2. 减少分析 prompt 的长度
3. 启用缓存机制
